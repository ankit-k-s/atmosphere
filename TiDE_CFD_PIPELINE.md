# TiDE + CFD Pipeline (focused on tide prediction)

This document summarizes the ML pipeline structure for tide (TiDE) prediction combined with CFD-derived features, and explains how TiDE + CFD are currently run in this repository. It intentionally focuses on TiDE-based tide prediction and treats TFT-related code as out-of-scope.

## Short answers (direct)

- ML pipeline structure: data ingestion -> preprocessing -> CFD feature generation -> Nixtla/TiDE model (history + futr exog) -> inference -> evaluation.
- How TiDE + CFD are currently run: CFD-derived features are generated by scripts (e.g., `physics_engine.py` / advection steps) and merged into the time-series dataset; `evaluate_tide.py` uses a saved Nixtla/TiDE model from `models/tide_global` and performs make_future_dataframe, prepares time/static exogenous features, predicts a 96-hour forecast, then merges predictions with the test vault for evaluation.

## Where to find the relevant code (key files)

- Main tide-related scripts: `atmosphere/train_tide.py`, `atmosphere/evaluate_tide.py`, `atmosphere/tide_grid_search.py`, `atmosphere/physics_engine.py`, `atmosphere/generate_96hr_digital_twin.py`.
- Data input for evaluation (example): `data/processed/tft/tide_ready_1hr.parquet` and vaulted test: `data/processed/tft/evaluation/test_holdout_set.parquet` (used by `evaluate_tide.py`).
- Model artifacts: `models/tide_global` (used by `evaluate_tide.py`), `models/tide_optimized_best`, `models/tide_temp`.
- TFT-related code (out-of-scope): `atmosphere/train_tft.py`, `atmosphere/prep_tft_dataset.py`, `models/tft/`.

> Note: Several scripts and datasets live in `atmosphere/` and `data/processed/`. This doc assumes the repository root is the workspace root.

## Pipeline (detailed, TiDE-focused)

Contract (2–4 bullets):
- Inputs: hourly time-series with station unique_id, timestamp, measured tide/pollution targets (y), static coords (latitude, longitude), and CFD-derived exogenous features (e.g., advected concentrations, wind components).
- Outputs: 96-hour forecasts per station (unique_id) with prediction quantiles (median used as primary) and an evaluation parquet containing real vs predicted values.
- Success criteria: Saved predictions aligned exactly to test vault timestamps and MAE/RMSE within an acceptable operational threshold.
- Error modes: missing static coords, mismatched unique_id formats, timezone-aware timestamp mismatches, or missing CFD features.

Steps:
1. Data ingestion
   - Load processed, hourly dataset (example: `data/processed/tft/tide_ready_1hr.parquet`). This file contains the historic series, static lat/lon, and precomputed CFD features (advection terms, etc.).

2. Preprocessing / Melt-format for Nixtla
   - If multiple target types exist, melt to Nixtla format (unique_id, ds, y). In `evaluate_tide.py` this happens in `prep_nixtla_format()` which also appends pollutant names to unique_id. For tide-only work, ensure `unique_id` reflects tide station id only.

3. CFD feature generation (external physics)
   - CFD/physics scripts compute advection and other spatial transport terms and write them into the processed dataset (e.g., functions in `physics_engine.py`, `build_advection_features.py`, `generate_96hr_digital_twin.py`).
   - These features are used as `hist_exog` and `futr_exog` for TiDE.

4. Make future dataframe (Nixtla / TiDE)
   - Use the TiDE-friendly Nixtla API (`NeuralForecast.make_future_dataframe`) to get the exact ds/unique_id combinations required for the forecast window.
   - Compute time-based features for futr_df (hour, day, weekday, month) and merge static spatial coords.

5. Prediction
   - Load pretrained TiDE model via `NeuralForecast.load(path=MODEL_DIR)`.
   - Call `nf.predict(df=df_train_val, futr_df=futr_df)` where `df_train_val` is history up to cutoff.
   - Select the desired prediction column (median/0.5 quantile). The repo's `evaluate_tide.py` dynamically finds the median column among numeric/quantile columns.

6. Postprocess & evaluation
   - Clip negative predictions to 0 if required by physics.
   - Merge with `test_holdout_set.parquet` on (`unique_id`, `ds`) to compute MAE/RMSE.
   - Save evaluation results to `data/processed/evaluation/tide_96hr_results.parquet`.

## How TiDE + CFD are currently run (step-by-step reproducible)

1. Prepare data with CFD features
   - Run the advection / CFD pipeline to generate/merge features into the processed Parquet. Typical scripts:
     - `atmosphere/build_advection_features.py` (or `physics_engine.py` / `generate_96hr_digital_twin.py`) — these compute transport/advection variables and write to `data/processed/...`.
   - Validate processed data contains expected columns (for example, `adv_in_PM2_5_ug_m3`, `adv_in_PM10_ug_m3`, `wind_x`, `wind_y`, `latitude`, `longitude`).

2. Train or load TiDE model
   - If training: run `atmosphere/train_tide.py` (script name implies TiDE training). This should create model artifacts under `models/tide_*`.
   - If using an already-trained model: ensure `models/tide_global` (or chosen folder) contains saved Nixtla/TiDE artifacts.

3. Run evaluation/inference
   - Example: run `atmosphere/evaluate_tide.py` (this is the current evaluation entrypoint). It performs:
     - Load the Nixtla/TiDE model (`NeuralForecast.load`).
     - Load historical dataset and transform to Nixtla format.
     - Recreate future grid via `nf.make_future_dataframe`.
     - Merge static features, call `nf.predict(...)` to get forecasts.
     - Extract median quantile column dynamically and rename to `prediction`.
     - Merge with `test_holdout_set.parquet` and compute MAE/RMSE by pollutant (or tide variable).
     - Save results to `data/processed/evaluation/tide_96hr_results.parquet`.

Commands (run from repo root):

```bash
python3 atmosphere/evaluate_tide.py
# or if you need training:
python3 atmosphere/train_tide.py
```

Notes about `evaluate_tide.py` behaviour (relevant to running):
- It expects `MODEL_DIR` to point at a Nixtla-managed model directory.
- It expects `INPUT_FILE` and `TEST_VAULT` to exist and to be Nixtla-friendly (ds as timestamp, unique_id, exog columns present).
- The script auto-detects the TiDE quantile column containing a '0.5' or '50' or 'median' substring; if none found, it falls back to the first TiDE column.

## Edge cases and operational considerations

- Missing static coords: `futr_df` joins on `unique_id`. If latitude/longitude is missing, predictions may be wrong or NaN.
- Timestamp timezone mismatches: ensure `ds` timestamps are timezone-naive or consistently timezone-aware.
- Quantile presence: TiDE outputs quantile columns; ensure your selection logic captures the desired quantile (current code uses heuristics).
- Negative physical values: the code clips negatives to 0 — confirm this is appropriate for your tide variable (it is appropriate for concentrations but might differ for tide heights).
